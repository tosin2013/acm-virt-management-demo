== Part 1 -- Why Right-Sizing Matters

=== Know

_In VMware environments, DRS (Distributed Resource Scheduler) automatically balances VM resource allocation across a cluster. When migrating to OpenShift Virtualization, organizations lose this built-in intelligence -- until now. RHACM 2.16 delivers GA right-sizing as the Kubernetes-native equivalent, using transparent Prometheus-based metrics instead of a proprietary black-box algorithm._

**Business Challenge:**
Organizations migrating from VMware consistently over-allocate VM resources "just in case." Without a data-driven feedback loop, teams request 4 GiB of memory for workloads that use 400 MiB, and 4 vCPUs for single-threaded services. This waste compounds across hundreds of VMs and dozens of clusters -- inflating infrastructure costs while starving other workloads of capacity.

**Technical Solution:**
RHACM 2.16 deploys right-sizing **automatically** when observability is enabled. No manual policy deployment is required. The system includes:

* *Recording rules policies* (`rs-prom-rules-policy` and `rs-virt-prom-rules-policy`) deployed via the `open-cluster-management-global-set` namespace to all managed clusters
* *Pre-built Grafana dashboards* in the "RightSizing Recommendation" and "ACM / OpenShift Virtualization" folders
* *Metrics allowlists* that forward `acm_rs_vm:*` recording rule results from managed clusters to the Hub's Thanos

[cols="1,1,2", options="header"]
|===
| Dashboard Level | Granularity | What It Shows

| Fleet Overview
| All clusters
| Total overestimation and underestimation counts across the managed fleet

| Namespace View
| Per-namespace
| Aggregated resource waste grouped by namespace with configurable time windows (15d / 30d)

| Per-VM Detail (Overestimation)
| Individual VM
| Time-series CPU and memory usage vs. requests, flagging over-provisioned VMs

| Per-VM Detail (Underestimation)
| Individual VM
| Time-series usage exceeding requests, flagging under-provisioned VMs at risk of OOM
|===

**Business Value:**

* *Zero deployment effort* -- right-sizing activates automatically with ACM Observability
* *Data-driven capacity planning* -- replace guesswork with actual usage data across the fleet
* *Cost optimization* -- identify and reclaim over-allocated resources systematically
* *Performance protection* -- flag under-provisioned VMs before users experience OOM kills or CPU throttling
* *VMware DRS replacement* -- GA Kubernetes-native alternative using open Prometheus standards

== Part 2 -- Explore the Built-In Right-Sizing Infrastructure

=== Show

NOTE: Ensure you are on the **Hub cluster terminal** and logged into OpenShift. If unsure, run `oc whoami` to verify. See xref:02-details.adoc#_connecting_to_clusters[Connecting to Clusters] for setup.

* Verify that the built-in right-sizing policies are already deployed and Compliant:
+
[source,role="execute"]
----
oc get policy -n open-cluster-management-global-set | grep rs-
----
+
Expected output shows both `rs-prom-rules-policy` and `rs-virt-prom-rules-policy` as `Compliant`.

* Inspect the VM-specific recording rules policy to understand what metrics are computed:
+
[source,role="execute"]
----
oc get policy rs-virt-prom-rules-policy -n open-cluster-management-global-set -o jsonpath='{.spec.policy-templates[0].objectDefinition.spec.object-templates[0].objectDefinition.spec.groups[0].rules[*].record}' | tr ' ' '\n'
----
+
These are the `acm_rs_vm:*` recording rules that power the dashboards:
+
** `acm_rs_vm:namespace:cpu_request:5m` -- vCPU count per VM
** `acm_rs_vm:namespace:cpu_usage:5m` -- actual CPU usage per VM
** `acm_rs_vm:namespace:memory_request:5m` -- requested memory per VM
** `acm_rs_vm:namespace:memory_usage:5m` -- actual memory usage per VM

* Verify the recording rules are producing data on the student cluster. SSH to the student cluster bastion:
+
[source,role="execute",subs="attributes+"]
----
{student_1_ssh_command}
----
+
Password: `{student_1_bastion_ssh_password}`

* Query the local Prometheus for VM right-sizing metrics:
+
[source,role="execute"]
----
oc exec -n openshift-monitoring prometheus-k8s-0 -c prometheus -- \
  promtool query instant http://localhost:9090 'acm_rs_vm:namespace:cpu_request:5m'
----
+
You should see entries for both the *fedora-web* and *win2019-install* VMs with their vCPU counts.

* Check the memory metrics for both VMs:
+
[source,role="execute"]
----
oc exec -n openshift-monitoring prometheus-k8s-0 -c prometheus -- \
  promtool query instant http://localhost:9090 'acm_rs_vm:namespace:memory_request:5m'
----
+
Notice the difference: *fedora-web* requests 4 GiB (a simple httpd server) while *win2019-install* requests 16 GiB (a Windows Server).

* View the 1-day aggregated recommendations:
+
[source,role="execute"]
----
oc exec -n openshift-monitoring prometheus-k8s-0 -c prometheus -- \
  promtool query instant http://localhost:9090 'acm_rs_vm:namespace:cpu_recommendation'
----
+
The recommendations include a 10% headroom margin (configurable via `rs-virt-config`). Compare these to the actual requests -- over-provisioned VMs will show a recommendation far below the current request.

* Return to the Hub terminal:
+
[source,role="execute"]
----
exit
----

== Part 4 -- Act on a Right-Sizing Recommendation (Fedora VM)

=== Know

_The right-sizing dashboards provide the data; the next step is acting on it. This demonstrates the complete feedback loop: observe overestimation → resize the VM → verify improvement. In production, this resize would be committed to Git and synced via ArgoCD._

=== Show

* The *fedora-web* VM is a classic overestimation case: it runs a simple httpd service but was allocated 1 vCPU and 2 GiB memory. The right-sizing data shows it actually uses ~9 millicores of CPU and ~377 MiB of memory.

* First, pause ArgoCD auto-sync on the Hub so the resize is not immediately reverted by GitOps self-heal:
+
[source,role="execute"]
----
oc patch application vm-fedora-web-student-1 -n openshift-gitops --type=merge -p '{"spec":{"syncPolicy":null}}'
----

* SSH to the managed cluster bastion:
+
[source,role="execute",subs="attributes+"]
----
{student_1_ssh_command}
----
+
Password: `{student_1_bastion_ssh_password}`

* Resize the Fedora VM memory from 2Gi down to 512Mi based on the recommendation:
+
[source,role="execute"]
----
oc patch virtualmachine fedora-web -n vm-demo-app --type=merge \
  -p '{"spec":{"template":{"spec":{"domain":{"resources":{"requests":{"memory":"512Mi"}}}}}}}'
----

* The VM will restart with the new memory allocation. Verify:
+
[source,role="execute"]
----
oc get vmi fedora-web -n vm-demo-app -o jsonpath='{.spec.domain.resources.requests.memory}'
echo
----
+
Expected output: `512Mi`

* Confirm the httpd service is still healthy after the resize:
+
[source,role="execute"]
----
curl -s -o /dev/null -w '%{http_code}' $(oc get vmi fedora-web -n vm-demo-app -o jsonpath='{.status.interfaces[0].ipAddress}'):8080
echo
----
+
Expected: `200` -- the web server is unaffected by the memory reduction.

* Return to the Hub terminal:
+
[source,role="execute"]
----
exit
----

* Re-enable ArgoCD auto-sync:
+
[source,role="execute"]
----
oc patch application vm-fedora-web-student-1 -n openshift-gitops --type=merge \
  -p '{"spec":{"syncPolicy":{"automated":{"prune":true,"selfHeal":true},"syncOptions":["CreateNamespace=true"]}}}'
----

[NOTE]
====
In a production workflow, the right-sizing change would be committed to the Git repository rather than applied directly. The ArgoCD sync-pause here demonstrates the feedback loop. In production, update the VM manifest in Git and let ArgoCD sync the change across the fleet.
====

== Part 5 -- Analyze the Windows VM Right-Sizing Data

=== Know

_Windows VMs are typically the most over-provisioned workloads in any fleet. Administrators allocate 8+ GiB "just for Windows" without measuring actual consumption. The right-sizing dashboards treat Windows VMs identically to Linux VMs -- providing the same transparent metrics._

=== Show

* SSH to the managed cluster bastion:
+
[source,role="execute",subs="attributes+"]
----
{student_1_ssh_command}
----
+
Password: `{student_1_bastion_ssh_password}`

* Check the current resource allocation for the Windows VM:
+
[source,role="execute"]
----
echo "=== Windows VM Resource Requests ==="
oc get vm win2019-install -n vm-demo-app -o jsonpath='CPU: {.spec.template.spec.domain.cpu.cores} cores{"\n"}Memory: {.spec.template.spec.domain.resources.requests.memory}{"\n"}'
echo
echo "=== Windows VM Actual Usage (from recording rules) ==="
oc exec -n openshift-monitoring prometheus-k8s-0 -c prometheus -- \
  promtool query instant http://localhost:9090 'acm_rs_vm:namespace:cpu_usage:5m{name="win2019-install"}'
echo
oc exec -n openshift-monitoring prometheus-k8s-0 -c prometheus -- \
  promtool query instant http://localhost:9090 'acm_rs_vm:namespace:memory_usage:5m{name="win2019-install"}'
----

* Compare the requested vs. actual usage:
+
[source,role="execute"]
----
echo "=== Recommendation (with 10% headroom) ==="
oc exec -n openshift-monitoring prometheus-k8s-0 -c prometheus -- \
  promtool query instant http://localhost:9090 'acm_rs_vm:namespace:cpu_recommendation{name="win2019-install"}'
echo
oc exec -n openshift-monitoring prometheus-k8s-0 -c prometheus -- \
  promtool query instant http://localhost:9090 'acm_rs_vm:namespace:memory_recommendation{name="win2019-install"}'
----
+
For a freshly installed Windows Server 2019 that is idle, expect the recommendation to be significantly lower than the 4-core / 16 GiB allocation. In production, Windows VMs running actual workloads would show higher utilization -- the key is measuring before resizing.

* Return to the Hub terminal:
+
[source,role="execute"]
----
exit
----

[IMPORTANT]
====
**Key insight for the customer:** Windows VMs are first-class citizens in the right-sizing framework. The same `acm_rs_vm:*` recording rules, dashboards, and recommendations apply regardless of guest OS. This eliminates the need for in-guest monitoring agents for capacity planning purposes.
====

== Part 6 -- Customize Right-Sizing Thresholds

=== Know

_The built-in right-sizing uses a default 110% recommendation margin (i.e., recommend 10% above peak usage). Organizations with stricter SLAs may want a higher margin, while cost-aggressive teams may accept a lower one. The `rs-virt-config` ConfigMap controls this behavior._

=== Show

* View the current right-sizing configuration on the Hub:
+
[source,role="execute"]
----
oc get cm rs-virt-config -n open-cluster-management-observability -o jsonpath='{.data.prometheusRuleConfig}' | cat
----
+
Key settings:
+
** `recommendationPercentage: 110` -- adds 10% headroom above observed peak usage
** `exclusionCriteria: ["openshift.*"]` -- excludes OpenShift system namespaces from right-sizing

* To change the recommendation margin (for example, to 125% for production VMs with stricter SLAs):
+
[source,role="execute"]
----
oc patch cm rs-virt-config -n open-cluster-management-observability --type=merge \
  -p '{"data":{"prometheusRuleConfig":"namespaceFilterCriteria:\n  inclusionCriteria: []\n  exclusionCriteria:\n  - openshift.*\nlabelFilterCriteria: []\nrecommendationPercentage: 125\n"}}'
----
+
[NOTE]
====
Changing the `recommendationPercentage` triggers a regeneration of the recording rules policy. Allow 5-10 minutes for the updated rules to propagate to all managed clusters and produce new recommendation values.
====

* Revert to the default 110% margin:
+
[source,role="execute"]
----
oc patch cm rs-virt-config -n open-cluster-management-observability --type=merge \
  -p '{"data":{"prometheusRuleConfig":"namespaceFilterCriteria:\n  inclusionCriteria: []\n  exclusionCriteria:\n  - openshift.*\nlabelFilterCriteria: []\nrecommendationPercentage: 110\n"}}'
----

== Key Takeaways

* RHACM 2.16 right-sizing is **built-in** -- it activates automatically with ACM Observability, no manual policy deployment required
* The `acm_rs_vm:*` recording rules and Grafana dashboards are deployed out of the box to all managed clusters
* Three-level drill-down: fleet → namespace → per-VM with overestimation and underestimation views
* Both Linux (Fedora) and Windows (Server 2019) VMs are analyzed identically -- no guest agent required
* The complete right-sizing loop: observe overestimation → resize → verify the workload is healthy → commit to Git
* Recommendation thresholds are configurable via `rs-virt-config` to match organizational risk tolerance
* This is the Kubernetes-native replacement for VMware DRS -- transparent, open-standards-based, and fleet-scale
