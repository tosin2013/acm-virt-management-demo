== Part 1 -- Beyond All-or-Nothing RBAC

=== Know

_Historically, managing a multicluster fleet with RHACM required broad administrative roles that exposed the entire infrastructure to potential mismanagement. The ClusterPermission API changes this fundamentally by enabling surgical, namespace-scoped permission distribution to managed clusters._

**Business Challenge:**
A VM operations team needs to start, stop, restart, and console into VMs on managed clusters. Today, this requires either:

* *cluster-admin on each managed cluster* -- too broad, exposes the entire cluster
* *Manual Role/RoleBinding creation on each cluster* -- does not scale, drifts over time

Neither option is acceptable for organizations with dozens of clusters and hundreds of VMs.

**Technical Solution:**
The `ClusterPermission` API (`clusterpermissions.rbac.open-cluster-management.io`) allows the Hub to:

1. Define a custom ClusterRole with exact kubevirt.io permissions
2. Automatically distribute the ClusterRole and ClusterRoleBinding to managed clusters
3. Maintain the lifecycle of distributed RBAC via ManifestWork synchronization

**Business Value:**

* *Surgical precision* -- grant exactly kubevirt.io operations, nothing else
* *Lifecycle management* -- permissions are continuously synchronized, not "set and forget"
* *Centralized audit* -- all permission grants are visible on the Hub
* *Group-based targeting* -- bind to a group, not individual users, for scalable team management

=== Show

* Review the ClusterPermission manifest before applying:
+
[source,role="execute"]
----
cat rbac/cluster-permission-vm-edit.yaml
----

* Note the key elements:
** `clusterRole.rules` defines exactly which kubevirt.io operations are permitted
** `clusterRoleBinding` creates a ClusterRoleBinding on the managed cluster for the `vm-operators` group
** The ClusterRole is auto-generated and distributed via ManifestWork synchronization
** The subject is a Group (`vm-operators`) -- not individual users, so team membership changes propagate automatically

== Part 2 -- Applying ClusterPermission

=== Know

_ClusterPermission resources are namespaced on the Hub -- the namespace name matches the managed cluster name. This is how ACM knows which managed cluster to target._

When the ClusterPermission is created, the ACM ClusterPermission controller:

1. Validates the requested roles exist on the target cluster
2. Generates the Role and RoleBinding manifests
3. Creates a ManifestWork resource to deploy them
4. Continuously monitors for drift and re-applies if needed

=== Show

* Apply the ClusterPermission to target `{student_cluster_name}`:
+
[source,role="execute"]
----
oc apply -f rbac/cluster-permission-vm-edit.yaml
----

* Verify the ClusterPermission was created:
+
[source,role="execute",subs="attributes+"]
----
oc get clusterpermissions -n {student_cluster_name}
----

* Check the ManifestWork was created to distribute the RBAC:
+
[source,role="execute",subs="attributes+"]
----
oc get manifestwork -n {student_cluster_name} | grep vm-editor
----

* SSH to the managed cluster bastion to verify the ClusterRole and ClusterRoleBinding were created:
+
[source,role="execute",subs="attributes+"]
----
{student_1_ssh_command}
----
+
Password: `{student_1_bastion_ssh_password}`

* Verify the distributed RBAC resources:
+
[source,role="execute"]
----
oc get clusterrole vm-editor-permission
oc get clusterrolebinding vm-editor-permission -o yaml | grep -A3 subjects
----

* Return to the Hub terminal:
+
[source,role="execute"]
----
exit
----

== Part 3 -- Demonstrating Scoped VM Access

=== Know

_The proof is in the access. A user in the `vm-operators` group can now manage VMs on the managed cluster -- and is denied access to everything else._

This is the exact capability enterprises need: their VM operations team can start, stop, restart, and console into VMs without seeing the underlying node configuration, network topology, or storage infrastructure. These access checks must run on the managed cluster where the ClusterPermission distributed the RBAC.

=== Show

* ClusterPermission distributes RBAC to the managed cluster, so the access checks must run there. SSH to the student cluster bastion:
+
[source,role="execute",subs="attributes+"]
----
{student_1_ssh_command}
----
+
Password: `{student_1_bastion_ssh_password}`

* As a vm-operators group member, demonstrate permitted actions:
+
[source,role="execute"]
----
oc auth can-i get virtualmachines.kubevirt.io -n vm-production --as-group=vm-operators --as=vm-user
----
+
Expected output: `yes`

+
[source,role="execute"]
----
oc auth can-i update virtualmachines/start.subresources.kubevirt.io -n vm-production --as-group=vm-operators --as=vm-user
----
+
Expected output: `yes`

* Now demonstrate denied actions -- the same user cannot access cluster-level resources:
+
[source,role="execute"]
----
oc auth can-i get nodes --as-group=vm-operators --as=vm-user
----
+
Expected output: `no`

+
[source,role="execute"]
----
oc auth can-i get secrets -n openshift-config --as-group=vm-operators --as=vm-user
----
+
Expected output: `no`

* Show that the user has no access outside the permitted namespaces:
+
[source,role="execute"]
----
oc auth can-i get virtualmachines.kubevirt.io -n kube-system --as-group=vm-operators --as=vm-user
----
+
Expected output: `no`

* Return to the Hub terminal:
+
[source,role="execute"]
----
exit
----

[NOTE]
====
**Point out to the customer:** "This operator can start, stop, restart, and console into any VM on this managed cluster. They cannot see a single node, secret, or cluster-level resource outside their granted permissions. And this permission is centrally managed from the Hub -- if you revoke the ClusterPermission, the RBAC disappears from all managed clusters within seconds."
====

== Key Takeaways

* ClusterPermission API replaces manual Role/RoleBinding management at scale
* Permissions are continuously synchronized via ManifestWork -- drift is automatically corrected
* VM operators get exactly the access they need -- kubevirt.io operations, nothing else
* Revoking access on the Hub propagates to all managed clusters automatically
* Group-based subjects mean team membership changes propagate without manifest updates
