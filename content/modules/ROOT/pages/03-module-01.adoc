== Part 1 -- Declarative VM Governance with ACM Policies

=== Know

_Traditional VM management relies on manual runbooks and per-cluster configuration. A single missed patch or misconfigured backup schedule across 15 clusters can result in data loss or compliance violations. RHACM eliminates this risk by treating VM governance as code._

**Business Challenge:**
Organizations running hundreds of VMs across multiple OpenShift clusters have no centralized way to enforce configuration standards. Each cluster team applies policies independently, leading to drift, inconsistent backup schedules, and blind spots in resource utilization.

**Technical Solution:**
RHACM uses `ConfigurationPolicy` resources to wrap kubevirt.io objects in a declarative governance layer. The policy engine continuously monitors managed clusters and automatically remediates drift -- if a required configuration is deleted or modified, ACM recreates it within seconds.

**Business Value:**

* *Automated compliance* -- policies map directly to NIST SP 800-53 controls (CM-2, CP-9)
* *Self-healing infrastructure* -- drift is corrected without human intervention
* *Targeted enforcement* -- policies apply only to clusters labeled `virtualization-enabled: true`

=== Show

* First, label the student cluster for virtualization targeting:
+
[source,role="execute"]
----
oc label managedcluster {student_cluster_name} virtualization-enabled=true --overwrite
----

* Verify that the managed clusters are labeled:
+
[source,role="execute"]
----
oc get managedclusters -l virtualization-enabled=true
----

* Create a ManagedClusterSetBinding so the Placement can select managed clusters:
+
[source,role="execute"]
----
cat <<'EOF' | oc apply -f -
apiVersion: cluster.open-cluster-management.io/v1beta2
kind: ManagedClusterSetBinding
metadata:
  name: global
  namespace: open-cluster-management-policies
spec:
  clusterSet: global
EOF
----

* Review the Placement resource that targets virtualization-enabled clusters:
+
[source,role="execute"]
----
oc get placement virt-enabled-clusters -n open-cluster-management-policies -o yaml
----

* Verify the PlacementDecision shows the expected clusters:
+
[source,role="execute"]
----
oc get placementdecision -n open-cluster-management-policies -l cluster.open-cluster-management.io/placement=virt-enabled-clusters -o yaml
----

== Part 2 -- VM Right-Sizing Metrics Policy

=== Know

_Right-sizing VMs is one of the primary reasons organizations adopted traditional hypervisor tools like VMware DRS. RHACM brings this capability to Kubernetes-native VM management by deploying Prometheus recording rules that continuously track CPU and memory utilization ratios._

The `vm-right-sizing-metrics` policy deploys two key components to every virtualization-enabled managed cluster:

* *PrometheusRule* with recording rules that calculate `kubevirt:vm_cpu_usage_ratio` and `kubevirt:vm_memory_usage_ratio`
* *Custom metrics allowlist* that ensures VM metrics are forwarded to the ACM Hub observability stack

These metrics power the fleet-wide Grafana dashboards demonstrated in Module 2.

=== Show

* Apply the VM right-sizing policy:
+
[source,role="execute"]
----
oc apply -f policies/vm-right-sizing-policy.yaml
oc apply -f policies/placement-binding.yaml
----

* Navigate to the ACM console at {openshift_console_url}

* Go to *Governance* → *Policies* and verify the `vm-right-sizing-metrics` policy shows *Compliant*

* SSH to the student cluster bastion to verify the resources on the managed cluster:
+
[source,role="execute"]
----
{student_1_ssh_command}
----
+
Password: `{student_1_bastion_ssh_password}`

* Verify the PrometheusRule was created:
+
[source,role="execute"]
----
oc get prometheusrule vm-right-sizing-rules -n openshift-monitoring -o yaml
----

* Verify the recording rules are producing data:
+
[source,role="execute"]

== Part 3 -- VM Backup Enforcement Policy

=== Know

_Business continuity for VM workloads requires automated, verified backups. The `vm-backup-enforcement` policy ensures that every virtualization-enabled cluster has OADP (OpenShift API for Data Protection) installed and a daily backup schedule configured for critical VM namespaces._

The policy enforces three layers of backup infrastructure:

1. *OADP Operator subscription* -- ensures the backup tooling is installed
2. *Backup schedule* -- daily 2:00 AM backups of `vm-production` and `vm-staging` namespaces with CSI snapshots
3. *DataProtectionApplication* -- configures Velero with the kubevirt plugin for VM-aware backups

If any of these components are deleted or modified, ACM automatically recreates them.

=== Show

* Apply the backup enforcement policy:
+
[source,role="execute"]
----
oc apply -f policies/vm-backup-enforcement-policy.yaml
----

* In the ACM console, navigate to *Governance* → *Policies* and locate `vm-backup-enforcement`

* Click the policy name to view compliance status per cluster

* SSH to the student cluster bastion to verify the resources on the managed cluster:
+
[source,role="execute"]
----
{student_1_ssh_command}
----
+
Password: `{student_1_bastion_ssh_password}`

* Verify the backup schedule exists:
+
[source,role="execute"]
----
oc get schedule -n openshift-adp
----

* *Simulate drift* -- delete the backup schedule and watch ACM remediate:
+
[source,role="execute"]
----
oc delete schedule vm-critical-backup -n openshift-adp
----

* Wait 30-60 seconds, then verify the schedule was automatically recreated:
+
[source,role="execute"]
----

----

* Return to the Hub terminal:
+
[source,role="execute"]
----
exit
----

[NOTE]
====
This self-healing behavior is the core value proposition: infrastructure drift is corrected automatically, eliminating the risk of missed backups due to human error or malicious deletion.
====

== Part 4 -- VM Network Isolation and Resource Guardrails

=== Know

_In a shared cluster environment, unrestricted network access between VM namespaces creates lateral movement risk. A compromised VM in `vm-staging` could reach production services in `vm-production`. Similarly, a VM created with extreme resource requests (e.g., 64Gi memory on a 32Gi node) can starve other workloads._

**Network Isolation Policy** (`vm-network-isolation`) enforces a default-deny NetworkPolicy in VM namespaces, allowing only:

* Same-namespace pod-to-pod traffic
* Monitoring scrape from `openshift-monitoring`
* DNS resolution via `openshift-dns`
* Outbound HTTP/HTTPS for package updates

**Resource Guardrails Policy** (`vm-resource-guardrails`) enforces a LimitRange in VM namespaces that prevents extreme resource requests:

* Min CPU: 500m, Max CPU: 8 cores
* Min Memory: 512Mi, Max Memory: 16Gi
* Default requests: 1 CPU, 1Gi memory

Both policies map to NIST SP 800-53 controls: SC-7 (Boundary Protection) and CM-7 (Least Functionality).

=== Show

* Apply the network isolation and resource guardrails policies:
+
[source,role="execute"]
----

oc apply -f policies/vm-resource-guardrails-policy.yaml
----

* Verify the policies are Compliant in the ACM console under *Governance* -> *Policies*

== Part 5 -- VM Eviction Strategy and Security Hardening

=== Know

_Non-migratable VMs with a `LiveMigrate` eviction strategy can block OpenShift upgrades and node maintenance indefinitely. The `vm-eviction-strategy` policy ensures every VM in the fleet is configured for graceful live migration during maintenance windows._

_Additionally, production server VMs rarely need serial console or graphics device access. Leaving these enabled expands the QEMU attack surface unnecessarily. The `vm-security-hardening` policy creates a `VirtualMachineClusterPreference` that disables these interfaces for server workloads._

**Eviction Strategy Policy** -- uses `remediationAction: inform` to detect VMs that lack the `LiveMigrate` eviction strategy without forcefully restarting them. Maps to NIST CP-2 (Contingency Plan).

**Security Hardening Policy** -- creates a `VirtualMachineClusterPreference` named `hardened-server-vm` that disables serial console and graphics device attachment. Maps to NIST SC-28 (Protection of Information at Rest) and CM-6 (Configuration Settings).

=== Show

* Apply the eviction strategy and security hardening policies:
+
[source,role="execute"]
----
oc apply -f policies/vm-eviction-strategy-policy.yaml
oc apply -f policies/vm-security-hardening-policy.yaml
----

* Verify the policies in the ACM console under *Governance* -> *Policies*

* SSH to the student cluster bastion to verify the hardened VM preference exists:
+
[source,role="execute"]
----
{student_1_ssh_command}
----
+
Password: `{student_1_bastion_ssh_password}`

* Verify the VirtualMachineClusterPreference was created:
+
[source,role="execute"]
----
oc get virtualmachineclusterpreferences hardened-server-vm -o yaml
----

* Return to the Hub terminal:
+
[source,role="execute"]
----

----

[NOTE]
====
When creating new VMs, reference the hardened preference to automatically inherit the security settings: `spec.preference.name: hardened-server-vm`. This provides consistent security posture across the fleet without manual configuration per VM.
====

== Key Takeaways

* ACM `ConfigurationPolicy` treats VM governance as declarative code
* Policies target only relevant clusters via `Placement` with label selectors
* Right-sizing metrics enable data-driven capacity planning without traditional hypervisor tools
* Backup enforcement guarantees business continuity with automated drift remediation
* Network isolation and resource guardrails prevent lateral movement and noisy-neighbor issues
* Eviction strategy enforcement protects upgrade and maintenance operations
* Security hardening reduces the QEMU attack surface for production VMs
* All policies map to recognized compliance frameworks (NIST SP 800-53)
