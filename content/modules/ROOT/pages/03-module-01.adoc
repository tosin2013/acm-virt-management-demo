== Part 1 -- Deploy a Fedora VM via GitOps

=== Know

_Before governing, monitoring, or securing virtual machines, you need to deploy them. This module gets VMs running on your managed clusters using two deployment methods: GitOps-driven Fedora VMs and ISO-based Windows Server installation._

The Fedora deployment uses an ArgoCD ApplicationSet with the `clusterDecisionResource` generator, which dynamically resolves which managed clusters to target based on RHACM Placement resources. Adding or removing clusters from the fleet automatically updates the application deployment -- no manual ArgoCD configuration required.

**Business Value:**

* *Fleet-scale deployment* -- a single ApplicationSet deploys VMs to every matching cluster
* *Dynamic targeting* -- cluster label changes automatically add or remove deployment targets
* *GitOps audit trail* -- the VM definition lives in Git, every change is a commit

=== Show

* First, label the student cluster so RHACM Placement resources can target it for virtualization workloads:
+
[source,role="execute",subs="attributes+"]
----
oc label managedcluster {student_cluster_name} virtualization-enabled=true --overwrite
----

* Verify the label was applied:
+
[source,role="execute"]
----
oc get managedclusters -l virtualization-enabled=true
----

* Create a namespace for the VM workload on the Hub:
+
[source,role="execute"]
----
oc create namespace vm-demo-app --dry-run=client -o yaml | oc apply -f -
----

* Configure the ACM-ArgoCD integration by creating a `ManagedClusterSetBinding`, `Placement`, and `GitOpsCluster` resource. This enables ArgoCD's `clusterDecisionResource` generator to discover managed clusters via ACM Placement:
+
[source,role="execute"]
----
cat <<'EOF' | oc apply -f -
apiVersion: cluster.open-cluster-management.io/v1beta2
kind: ManagedClusterSetBinding
metadata:
  name: global
  namespace: openshift-gitops
spec:
  clusterSet: global
---
apiVersion: cluster.open-cluster-management.io/v1beta1
kind: Placement
metadata:
  name: virt-enabled-clusters
  namespace: openshift-gitops
spec:
  predicates:
    - requiredClusterSelector:
        labelSelector:
          matchLabels:
            virtualization-enabled: "true"
---
apiVersion: apps.open-cluster-management.io/v1beta1
kind: GitOpsCluster
metadata:
  name: gitops-cluster
  namespace: openshift-gitops
spec:
  argoServer:
    cluster: local-cluster
    argoNamespace: openshift-gitops
  placementRef:
    kind: Placement
    apiGroup: cluster.open-cluster-management.io
    name: virt-enabled-clusters
    namespace: openshift-gitops
EOF
----

* Apply the ApplicationSet that deploys a Fedora VM to all virtualization-enabled clusters:
+
[source,role="execute"]
----
cat <<'EOF' | oc apply -f -
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: vm-fedora-web
  namespace: openshift-gitops
spec:
  generators:
    - clusterDecisionResource:
        configMapRef: acm-placement
        labelSelector:
          matchLabels:
            cluster.open-cluster-management.io/placement: virt-enabled-clusters
        requeueAfterSeconds: 180
  template:
    metadata:
      name: 'vm-fedora-web-{{name}}'
    spec:
      project: default
      source:
        repoURL: 'https://github.com/tosin2013/acm-virt-management-demo.git'
        targetRevision: main
        path: examples/vm-fedora-web
      destination:
        server: '{{server}}'
        namespace: vm-demo-app
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
EOF
----

* Wait for ArgoCD to sync the application to the managed clusters:
+
[source,role="execute"]
----
oc get applications.argoproj.io -n openshift-gitops
----
+
You should see an application named `vm-fedora-web-student-1` with Sync Status `Synced` and Health Status `Healthy`.
+
[TIP]
====
If `No resources found` appears, the GitOpsCluster controller may need a few minutes to register newly imported managed clusters. Wait 2-3 minutes and re-run the command. You can check progress with:

[source,bash]
----
oc get gitopscluster gitops-cluster -n openshift-gitops -o jsonpath='{.status}' | python3 -m json.tool
----
====

== Part 2 -- Deploy a Windows Server 2019 VM

=== Know

_Windows Server workloads represent a significant portion of enterprise VM estates. OpenShift Virtualization supports Windows VMs natively, and this part demonstrates the ISO-based installation workflow -- the same approach administrators use with traditional hypervisors, but integrated into the Kubernetes ecosystem._

The workflow uses four components:

* *HTTP File Server* -- an in-cluster file server (deployed on the Hub) where you upload the Windows ISO via a web UI
* *CDI DataVolumes* -- Kubernetes resources that import the ISO from the file server into a PVC for the VM to boot from
* *VirtIO drivers* -- Red Hat provides a container disk with Windows VirtIO drivers, attached as a secondary CD-ROM during installation
* *Autounattend / Sysprep* -- XML answer files that automate the Windows installation and generalize the image for cloning

**Business Value:**

* *Fully automated install* -- the `autounattend.xml` answer file eliminates all interactive prompts, installs VirtIO drivers, enables RDP, and installs the QEMU Guest Agent automatically
* *Golden-image ready* -- a built-in sysprep step generalizes the finished VM so the disk can be cloned as a reusable template
* *In-cluster hosting* -- ISOs are served from within the cluster, no external file shares needed
* *Automated pipeline* -- Tekton pipelines can automate the entire image build process

=== Show

* Open the HTTP File Server UI in your browser:
+
----
https://httpd-server-httpd-server.{openshift_cluster_ingress_domain}
----
+
Log in with your OpenShift credentials when prompted by the OAuth login page.

* Upload the Windows Server 2019 ISO using the drag-and-drop interface. Name the file `win2k19.iso`.

[TIP]
====
The file server uses OpenShift OAuth for external access. Any user registered with the cluster's identity providers can log in. Internal cluster access (used by DataVolumes) does not require authentication.
====

* Create a DataVolume to import the Windows ISO from the file server into a PVC:
+
[source,role="execute"]
----
cat <<'EOF' | oc apply -f -
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: win2019-iso
  namespace: vm-demo-app
  annotations:
    cdi.kubevirt.io/storage.bind.immediate.requested: "true"
spec:
  source:
    http:
      url: "http://httpd-server.httpd-server.svc.cluster.local:8080/files/win2k19.iso"
  pvc:
    accessModes: [ReadWriteOnce]
    storageClassName: gp3-csi
    resources:
      requests:
        storage: 7Gi
EOF
----

* Create a blank disk DataVolume for the Windows installation target:
+
[source,role="execute"]
----
cat <<'EOF' | oc apply -f -
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: win2019-disk
  namespace: vm-demo-app
  annotations:
    cdi.kubevirt.io/storage.bind.immediate.requested: "true"
spec:
  source:
    blank: {}
  pvc:
    accessModes: [ReadWriteOnce]
    storageClassName: gp3-csi
    resources:
      requests:
        storage: 60Gi
EOF
----

* Wait for the ISO import to complete:
+
[source,role="execute"]
----
oc get datavolume win2019-iso -n vm-demo-app -w
----
+
The DataVolume progresses through `ImportScheduled` → `ImportInProgress` → `Succeeded`. This may take several minutes depending on ISO size.

* Create ConfigMaps from the `autounattend.xml` and `sysprep-unattend.xml` answer files. These automate the entire Windows installation -- no manual prompts:
+
[source,role="execute"]
----
curl -sL https://raw.githubusercontent.com/tosin2013/acm-virt-management-demo/main/examples/vm-win2019/autounattend.xml -o /tmp/autounattend.xml
oc create configmap win2019-autounattend -n vm-demo-app \
  --from-file=autounattend.xml=/tmp/autounattend.xml \
  --dry-run=client -o yaml | oc apply -f -
----
+
[source,role="execute"]
----
curl -sL https://raw.githubusercontent.com/tosin2013/acm-virt-management-demo/main/examples/vm-win2019/sysprep-unattend.xml -o /tmp/sysprep-unattend.xml
oc create configmap win2019-sysprep -n vm-demo-app \
  --from-file=sysprep-unattend.xml=/tmp/sysprep-unattend.xml \
  --dry-run=client -o yaml | oc apply -f -
----
+
[TIP]
====
Alternatively, clone the repo and create from local files:

[source,bash]
----
git clone https://github.com/tosin2013/acm-virt-management-demo.git
oc create configmap win2019-autounattend -n vm-demo-app \
  --from-file=autounattend.xml=acm-virt-management-demo/examples/vm-win2019/autounattend.xml
oc create configmap win2019-sysprep -n vm-demo-app \
  --from-file=sysprep-unattend.xml=acm-virt-management-demo/examples/vm-win2019/sysprep-unattend.xml
----
====

* Create the Windows installation VM with VirtIO drivers and autounattend attached:
+
[source,role="execute"]
----
cat <<'EOF' | oc apply -f -
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: win2019-install
  namespace: vm-demo-app
  labels:
    app: windows-install
spec:
  runStrategy: Always
  template:
    metadata:
      labels:
        kubevirt.io/domain: win2019-install
    spec:
      domain:
        cpu:
          cores: 4
        devices:
          disks:
            - disk:
                bus: virtio
              name: installdisk
              bootOrder: 2
            - cdrom:
                bus: sata
              name: iso
              bootOrder: 1
            - cdrom:
                bus: sata
              name: virtiocontainerdisk
            - disk:
                bus: virtio
              name: sysprep
          interfaces:
            - masquerade: {}
              name: default
          inputs:
            - type: tablet
              bus: usb
              name: tablet
        resources:
          requests:
            memory: 8Gi
        machine:
          type: q35
        features:
          acpi: {}
          hyperv:
            relaxed: {}
            vapic: {}
            spinlocks:
              spinlocks: 8191
      networks:
        - name: default
          pod: {}
      volumes:
        - name: installdisk
          persistentVolumeClaim:
            claimName: win2019-disk
        - name: iso
          persistentVolumeClaim:
            claimName: win2019-iso
        - name: virtiocontainerdisk
          containerDisk:
            image: registry.redhat.io/container-native-virtualization/virtio-win-rhel9@sha256:7e06e1f52a434d4602657c920144504fbaed955d0998535bdf345716355ce83a
        - name: sysprep
          sysprep:
            configMap:
              name: win2019-autounattend
EOF
----

[NOTE]
====
The `sysprep` volume uses KubeVirt's built-in sysprep disk support. It mounts the `autounattend.xml` ConfigMap as a virtual floppy drive (`A:\`), which Windows Setup reads automatically during installation. This eliminates all interactive prompts:

* VirtIO storage and network drivers are loaded automatically
* Windows Server 2019 Standard (Desktop Experience) is selected
* Disk partitioning, hostname (`win2019-demo`), and timezone (UTC) are configured
* Administrator password is set to `R3dH4tD3mo!`
* RDP and WinRM are enabled
* QEMU Guest Agent is installed at first login
====

* After Windows finishes installing and auto-logs in as Administrator (fully automated), the VM is ready. Verify the QEMU Guest Agent is reporting:
+
[source,role="execute"]
----
oc wait --for=condition=AgentConnected vmi/win2019-install -n vm-demo-app --timeout=1800s
----
+
NOTE: A full automated Windows install from ISO takes 15-30 minutes. The command above waits up to 30 minutes for the QEMU Guest Agent to connect after Windows Setup completes and runs the FirstLogonCommands.
+
[TIP]
====
If the wait times out, the VM may still be running correctly -- the Guest Agent installation requires outbound internet access from the VM pod network to download the MSI package. You can verify the VM is running with `oc get vmi win2019-install -n vm-demo-app` and check the console via the OpenShift web UI. The Guest Agent is not required for the remaining modules.
====

==== Generalize with Sysprep (Optional -- for golden-image cloning)

If you want to turn this installed VM into a reusable golden image, run sysprep to strip machine-specific identity (SID, hostname). This step is *optional* for the demo but required in production image-building workflows.

* Stop the VM and swap the autounattend ConfigMap for the sysprep answer file:
+
[source,role="execute"]
----
virtctl stop win2019-install -n vm-demo-app
----
+
[source,role="execute"]
----
oc patch virtualmachine win2019-install -n vm-demo-app --type=json -p='[
  {"op": "remove", "path": "/spec/template/spec/volumes/3"},
  {"op": "remove", "path": "/spec/template/spec/domain/devices/disks/3"}
]'
----
+
[source,role="execute"]
----
oc patch virtualmachine win2019-install -n vm-demo-app --type=json -p='[
  {"op": "add", "path": "/spec/template/spec/domain/devices/disks/-",
   "value": {"disk": {"bus": "virtio"}, "name": "sysprep"}},
  {"op": "add", "path": "/spec/template/spec/volumes/-",
   "value": {"name": "sysprep", "sysprep": {"configMap": {"name": "win2019-sysprep"}}}}
]'
----

* Start the VM and open the console. Windows will boot to the desktop:
+
[source,role="execute"]
----
virtctl start win2019-install -n vm-demo-app
----

* From the VM console (or via RDP), run sysprep:
+
[source]
----
C:\Windows\System32\Sysprep\sysprep.exe /generalize /oobe /shutdown /unattend:A:\sysprep-unattend.xml
----
+
The VM will generalize itself and shut down. The resulting `win2019-disk` PVC is now a clean golden image that can be cloned for new VMs without SID conflicts.

== Part 3 -- Verify VM Deployments

=== Know

_With both Fedora and Windows VMs deployed, verify they are running correctly. The Fedora VM should be fully operational via GitOps, and the Windows VM should have completed its automated installation._

=== Show

* Check the Fedora VM status on the managed cluster:
+
[source,role="execute",subs="attributes+"]
----
{student_1_ssh_command}
----
+
Password: `{student_1_bastion_ssh_password}`

+
[source,role="execute"]
----
oc get virtualmachines -n vm-demo-app
oc get virtualmachineinstances -n vm-demo-app
----

* Verify the Fedora VM is running and accessible:
+
[source,role="execute"]
----
oc get vmi fedora-web -n vm-demo-app -o jsonpath='{.status.phase}'
----
+
Expected output: `Running`

* Return to the Hub terminal:
+
[source,role="execute"]
----
exit
----

* Check the Windows VM status on the Hub (it runs locally since the DataVolumes were created on the Hub):
+
[source,role="execute"]
----
oc get virtualmachine win2019-install -n vm-demo-app
oc get datavolume -n vm-demo-app
----

* Access the Windows VM desktop through the OpenShift console:
** Navigate to {openshift_console_url}
** Go to *Virtualization* → *VirtualMachines* → select `win2019-install` → *Console*
** You should see the Windows Server 2019 desktop (auto-logged in as Administrator)

* Verify RDP is enabled and the QEMU Guest Agent is running (from the VM console):
+
[source]
----
powershell -Command "Get-Service QEMU-GA | Select-Object Status"
----
+
Expected output: `Running`

== Key Takeaways

* GitOps-driven VM deployment with ApplicationSets provides fleet-scale automation
* The `clusterDecisionResource` generator dynamically targets clusters via ACM Placement
* Windows VMs use the same ISO-based installation workflow as traditional hypervisors
* `autounattend.xml` eliminates all interactive prompts -- VirtIO drivers, RDP, Guest Agent are configured automatically
* Sysprep generalization produces reusable golden images without SID conflicts
* CDI DataVolumes handle ISO import and disk provisioning declaratively
* In-cluster file hosting eliminates the need for external NFS/SMB shares
* Both Linux and Windows VMs are first-class citizens on OpenShift Virtualization
