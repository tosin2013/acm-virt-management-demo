== Part 1 -- Set Up ACM-ArgoCD Integration

=== Know

_Before deploying VMs via GitOps, you need to integrate RHACM's cluster inventory with OpenShift GitOps (ArgoCD). This enables ArgoCD to discover managed clusters through ACM Placement resources and deploy workloads to them automatically._

The integration follows four steps:

* *ManagedClusterSet* -- groups clusters that should be visible to ArgoCD
* *ManagedClusterSetBinding* -- grants the `openshift-gitops` namespace access to the cluster set
* *Placement* -- selects clusters from the set using label-based predicates
* *GitOpsCluster* -- registers the selected clusters as ArgoCD deployment targets
* *application-manager addon* -- provides the credentials ArgoCD needs to reach managed clusters

**Business Value:**

* *Single pane of glass* -- ArgoCD sees all managed clusters without manual secret management
* *Dynamic fleet membership* -- label a new cluster and it automatically becomes an ArgoCD target
* *Separation of concerns* -- cluster admins manage labels, GitOps teams manage applications

=== Show

* Create a dedicated ManagedClusterSet for virtualization workloads:
+
[source,role="execute"]
----
cat <<'EOF' | oc apply -f -
apiVersion: cluster.open-cluster-management.io/v1beta2
kind: ManagedClusterSet
metadata:
  name: virt-clusters
EOF
----

* Label the student cluster into the set and mark it for virtualization workloads:
+
[source,role="execute",subs="attributes+"]
----
oc label managedcluster {student_cluster_name} \
  cluster.open-cluster-management.io/clusterset=virt-clusters \
  virtualization-enabled=true --overwrite
----

* Verify the cluster is in the set:
+
[source,role="execute"]
----
oc get managedclusters -l cluster.open-cluster-management.io/clusterset=virt-clusters
----

* Install the `application-manager` addon on the student cluster. This provides the credentials ArgoCD needs to deploy workloads to managed clusters:
+
[source,role="execute",subs="attributes+"]
----
cat <<'EOF' | oc apply -f -
apiVersion: addon.open-cluster-management.io/v1alpha1
kind: ManagedClusterAddOn
metadata:
  name: application-manager
  namespace: {student_cluster_name}
spec:
  installNamespace: open-cluster-management-agent-addon
EOF
----

* Create the ManagedClusterSetBinding, Placement, and GitOpsCluster resources to wire RHACM into ArgoCD:
+
[source,role="execute"]
----
cat <<'EOF' | oc apply -f -
apiVersion: cluster.open-cluster-management.io/v1beta2
kind: ManagedClusterSetBinding
metadata:
  name: virt-clusters
  namespace: openshift-gitops
spec:
  clusterSet: virt-clusters
---
apiVersion: cluster.open-cluster-management.io/v1beta1
kind: Placement
metadata:
  name: virt-enabled-clusters
  namespace: openshift-gitops
spec:
  clusterSets:
    - virt-clusters
  predicates:
    - requiredClusterSelector:
        labelSelector:
          matchLabels:
            virtualization-enabled: "true"
---
apiVersion: apps.open-cluster-management.io/v1beta1
kind: GitOpsCluster
metadata:
  name: gitops-cluster
  namespace: openshift-gitops
spec:
  argoServer:
    cluster: local-cluster
    argoNamespace: openshift-gitops
  placementRef:
    kind: Placement
    apiVersion: cluster.open-cluster-management.io/v1beta1
    name: virt-enabled-clusters
EOF
----

* Wait for the GitOpsCluster to register the managed cluster with ArgoCD (this may take 30-60 seconds):
+
[source,role="execute"]
----
oc get gitopscluster gitops-cluster -n openshift-gitops -o jsonpath='{.status.message}'
echo
----
+
Expected output: `Added managed clusters [student-1] to gitops namespace openshift-gitops`
+
[TIP]
====
If it still shows `[]`, wait 30 seconds and re-run. The `application-manager` addon needs time to create the cluster credentials secret. You can check progress with:

[source,bash]
----
oc get secrets -n openshift-gitops -l argocd.argoproj.io/secret-type=cluster --no-headers
----

You should see a secret named `{student_cluster_name}-application-manager-cluster-secret`.
====

== Part 2 -- Deploy a Fedora VM via GitOps

=== Know

_With ArgoCD integrated, you can now deploy VMs using an ApplicationSet. The Fedora deployment uses the `clusterDecisionResource` generator, which dynamically resolves which managed clusters to target based on RHACM Placement decisions. Adding or removing clusters from the fleet automatically updates the application deployment -- no manual ArgoCD configuration required._

**Business Value:**

* *Fleet-scale deployment* -- a single ApplicationSet deploys VMs to every matching cluster
* *Dynamic targeting* -- cluster label changes automatically add or remove deployment targets
* *GitOps audit trail* -- the VM definition lives in Git, every change is a commit

=== Show

* Apply the ApplicationSet that deploys a Fedora VM to all virtualization-enabled clusters:
+
[source,role="execute"]
----
cat <<'EOF' | oc apply -f -
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: vm-fedora-web
  namespace: openshift-gitops
spec:
  generators:
    - clusterDecisionResource:
        configMapRef: acm-placement
        labelSelector:
          matchLabels:
            cluster.open-cluster-management.io/placement: virt-enabled-clusters
        requeueAfterSeconds: 180
  template:
    metadata:
      name: 'vm-fedora-web-{{name}}'
    spec:
      project: default
      source:
        repoURL: 'https://github.com/tosin2013/acm-virt-management-demo.git'
        targetRevision: main
        path: examples/vm-fedora-web
      destination:
        server: '{{server}}'
        namespace: vm-demo-app
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
EOF
----

* Wait for ArgoCD to sync the application:
+
[source,role="execute"]
----
oc get applications.argoproj.io -n openshift-gitops
----
+
You should see an application named `vm-fedora-web-{student_cluster_name}` with Sync Status `Synced` and Health Status `Healthy`.
+
[TIP]
====
If `No resources found` appears, wait 1-2 minutes for the `clusterDecisionResource` generator to pick up the PlacementDecision. Re-run the command until the application appears.
====

== Part 3 -- Deploy a Windows Server 2019 VM via GitOps

=== Know

_Windows Server workloads represent a significant portion of enterprise VM estates. OpenShift Virtualization supports Windows VMs natively. This part demonstrates deploying a Windows VM entirely through GitOps -- the VM definition, DataVolumes, and answer files all live in Git and are synced by ArgoCD._

The workflow uses:

* *CDI DataVolumes* -- import the Windows ISO from the in-cluster file server and provision a blank install disk
* *VirtIO drivers* -- Red Hat provides a container disk with Windows VirtIO drivers, attached as a secondary CD-ROM
* *Autounattend* -- an XML answer file that automates the entire Windows installation with no interactive prompts
* *ArgoCD Application* -- deploys all resources to the student cluster in a single sync

[NOTE]
====
*Admin pre-requisite:* The Windows Server 2019 ISO must be uploaded to the HTTP File Server before this step. If you haven't done this yet, open the file server UI at `https://httpd-server-httpd-server.{student_1_ingress_domain}`, log in with OpenShift OAuth, and upload the ISO as `win2k19.iso`.

You can verify the ISO is available:

[source,bash]
----
components/httpd-fileserver/validate-iso.sh
----
====

**Business Value:**

* *Fully automated install* -- `autounattend.xml` eliminates all interactive prompts, installs VirtIO drivers, enables RDP, and installs the QEMU Guest Agent
* *Golden-image ready* -- a built-in sysprep step generalizes the finished VM for cloning
* *GitOps-managed* -- the entire VM stack (disk provisioning, answer files, VM definition) is version-controlled and auditable
* *In-cluster hosting* -- ISOs are served from within the cluster, no external file shares needed

=== Show

* Apply an ArgoCD Application that deploys the Windows VM stack (DataVolumes, ConfigMaps, and VirtualMachine) to the student cluster:
+
[source,role="execute",subs="attributes+"]
----
cat <<'EOF' | oc apply -f -
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: vm-win2019-{student_cluster_name}
  namespace: openshift-gitops
spec:
  project: default
  source:
    repoURL: 'https://github.com/tosin2013/acm-virt-management-demo.git'
    targetRevision: main
    path: examples/vm-win2019
  destination:
    server: 'https://cluster-proxy-addon-user.multicluster-engine.svc.cluster.local:9092/{student_cluster_name}'
    namespace: vm-demo-app
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
EOF
----

* Watch ArgoCD sync the Windows VM resources:
+
[source,role="execute"]
----
oc get applications.argoproj.io -n openshift-gitops
----
+
You should see both `vm-fedora-web-{student_cluster_name}` and `vm-win2019-{student_cluster_name}` applications. The Windows app will show `Synced` once all resources are created on the student cluster.

[NOTE]
====
The `autounattend.xml` in the ConfigMap automates the full Windows installation:

* VirtIO storage and network drivers are loaded automatically
* Windows Server 2019 Standard (Desktop Experience) is selected
* Disk partitioning, hostname (`win2019-demo`), and timezone (UTC) are configured
* Administrator password is set to `R3dH4tD3mo!`
* RDP and WinRM are enabled
* QEMU Guest Agent is installed at first login

The full installation takes 15-30 minutes from ISO boot to desktop.
====

==== Generalize with Sysprep (Optional -- for golden-image cloning)

If you want to turn this installed VM into a reusable golden image, run sysprep to strip machine-specific identity (SID, hostname). This step is *optional* for the demo but required in production image-building workflows.

* SSH to the student cluster:
+
[source,role="execute",subs="attributes+"]
----
{student_1_ssh_command}
----
+
Password: `{student_1_bastion_ssh_password}`

* Stop the VM and swap the autounattend ConfigMap for the sysprep answer file:
+
[source,role="execute"]
----
virtctl stop win2019-install -n vm-demo-app
----
+
[source,role="execute"]
----
oc patch virtualmachine win2019-install -n vm-demo-app --type=json -p='[
  {"op": "remove", "path": "/spec/template/spec/volumes/3"},
  {"op": "remove", "path": "/spec/template/spec/domain/devices/disks/3"}
]'
----
+
[source,role="execute"]
----
oc patch virtualmachine win2019-install -n vm-demo-app --type=json -p='[
  {"op": "add", "path": "/spec/template/spec/domain/devices/disks/-",
   "value": {"disk": {"bus": "virtio"}, "name": "sysprep"}},
  {"op": "add", "path": "/spec/template/spec/volumes/-",
   "value": {"name": "sysprep", "sysprep": {"configMap": {"name": "win2019-sysprep"}}}}
]'
----

* Start the VM and open the console. Windows will boot to the desktop:
+
[source,role="execute"]
----
virtctl start win2019-install -n vm-demo-app
----

* From the VM console (or via RDP), run sysprep:
+
[source]
----
C:\Windows\System32\Sysprep\sysprep.exe /generalize /oobe /shutdown /unattend:A:\sysprep-unattend.xml
----
+
The VM will generalize itself and shut down. The resulting `win2019-disk` PVC is now a clean golden image that can be cloned for new VMs without SID conflicts.

* Return to the Hub terminal:
+
[source,role="execute"]
----
exit
----

== Key Takeaways

* ACM-ArgoCD integration requires a custom ManagedClusterSet, explicit cluster labeling, and the `application-manager` addon
* The `clusterDecisionResource` generator dynamically targets clusters via ACM Placement -- label changes automatically update targets
* Both Linux and Windows VMs can be deployed entirely through GitOps (definitions in Git, synced by ArgoCD)
* `autounattend.xml` eliminates all interactive Windows prompts -- VirtIO drivers, RDP, Guest Agent are configured automatically
* CDI DataVolumes handle ISO import and disk provisioning declaratively
* Sysprep generalization produces reusable golden images without SID conflicts
* In-cluster file hosting eliminates the need for external NFS/SMB shares
