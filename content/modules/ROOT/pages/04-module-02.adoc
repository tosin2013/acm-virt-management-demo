== Part 1 -- Inspecting the Topology Graph

=== Know

_Application Topology is the visual proof that everything works together. After deploying VM-based applications via GitOps in Module 1, the topology view shows the complete dependency graph -- which components run where, their real-time status, and how they relate to each other._

**Business Challenge:**
In a multicluster environment, understanding "where is my application running and is it healthy?" requires correlating information across multiple cluster consoles, ArgoCD dashboards, and CLI sessions. This is error-prone and slow.

**Technical Solution:**
RHACM's Application Topology view interprets ArgoCD `Application` and `ApplicationSet` objects to render an interactive graph that:

* Shows all application components across Hub and managed clusters
* Displays real-time status (healthy, degraded, missing)
* Enables remote log retrieval directly from the topology UI
* Works for both container-based and VM-based workloads

**Business Value:**

* Visual validation of GitOps deployments -- see at a glance if everything landed correctly
* Remote diagnostics without kubeconfig access -- fetch logs through the UI
* Cross-cluster dependency mapping -- understand how services connect across clusters

=== Show

* Navigate to the ACM console at {openshift_console_url}

* Go to *Applications* in the left navigation

* Observe the list of managed applications -- the `vm-fedora-web` ApplicationSet deployed in Module 1 should be visible

image::acm-applications-list.png[ACM Applications list showing vm-fedora-web]

* Select `vm-fedora-web` to open the topology graph

image::acm-topology-graph.png[ACM Topology graph showing VM components across clusters]

* The topology graph displays:
** The ApplicationSet on the Hub
** The Application instances on each managed cluster
** The VirtualMachine resources and their status
** Associated Services, Routes, and DataVolumes

* Click on a VirtualMachine node in the graph to view:
** Resource details (CPU, memory requests)
** Current phase (Running, Stopped, Migrating)
** Events timeline

* Filter the topology by clicking the search/filter icon and typing `vm-fedora-web` to isolate just this application's components

* Toggle between *Graph view* and *List view* to show different perspectives on the same deployment

== Part 2 -- Remote Diagnostics via Topology

=== Know

_The topology view is the "proof screenshot" for the customer -- it visually confirms that the entire GitOps pipeline works end-to-end, from Git commit to running VM across multiple clusters._

A critical capability to highlight: *remote log retrieval*. Level-1 operators can inspect VM pod logs and diagnose issues (CrashLoopBackOff, boot failures, network misconfigurations) directly from the topology view -- without needing kubeconfig access to the managed cluster. This reinforces the zero-trust operational model from Module 6.

=== Show

* In the topology graph, locate a VirtualMachine node and click on its virt-launcher pod

* Click *View Logs* on the VM's virt-launcher pod to retrieve remote logs:

image::acm-topology-vm-logs.png[Remote log retrieval from VM topology view]

[NOTE]
====
Emphasize to the customer: "You just retrieved logs from a VM running on a remote cluster without having any kubeconfig or direct CLI access to that cluster. The ACM Hub proxied the request securely."
====

== Key Takeaways

* Application Topology provides visual validation of multicluster GitOps deployments
* The view works identically for container and VM-based workloads
* Remote log retrieval enables zero-trust diagnostics without per-cluster credentials
* ApplicationSets with `clusterDecisionResource` dynamically target clusters via ACM Placement
* The topology graph spans Hub and managed clusters in a single interactive view
