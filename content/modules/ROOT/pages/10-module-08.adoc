== Part 1 -- Why Cluster Destruction Prevention Matters

=== Know

_The most catastrophic risk in a hub-and-spoke architecture is the destruction of managed clusters from the Hub. A compromised credential, a fat-fingered CLI command, or a rogue insider with the `delete` verb on `managedclusters` can take down production workloads across an entire region in seconds._

**Business Challenge:**
The default `open-cluster-management:cluster-manager-admin` role grants `create`, `read`, `update`, and `delete` on all cluster lifecycle resources. This is the equivalent of giving every building manager a demolition permit -- technically correct, but operationally catastrophic.

**Technical Solution:**
This module deploys two layers of defense:

1. **Non-destructive ClusterRole** -- a custom `rhacm-nondestructive-operator` ClusterRole that explicitly omits the `delete` verb for all cluster lifecycle API groups
2. **ValidatingAdmissionPolicy** -- a Kubernetes-native admission policy that denies DELETE operations on cluster lifecycle resources for all non-admin users, regardless of their RBAC bindings

[cols="1,1,1", options="header"]
|===
| API Resource | Standard Admin Verbs | Non-Destructive Verbs

| `managedclusters`
| get, list, watch, create, update, delete
| get, list, watch, update, patch

| `clusterdeployments`
| get, list, watch, create, update, delete
| get, list, watch, update, patch

| `clusterclaims` / `clusterpools`
| get, list, watch, create, update, delete
| get, list, watch, update, patch

| `managedclustersets`
| get, list, watch, create, update, delete
| get, list, watch, update, patch

| `managedclustersetbindings`
| get, list, watch, create, update, delete
| get, list, watch, update, patch
|===

[IMPORTANT]
====
**Why two layers?** Kubernetes RBAC is _additive_ -- permissions from all bound ClusterRoles are unioned together. A user with both the non-destructive role AND a ManagedClusterSet admin binding would still have `delete` via the admin binding. The ValidatingAdmissionPolicy acts as a second gate: even if RBAC grants `delete`, the admission policy denies the API call before it reaches the resource.
====

**Business Value:**

* *Defense in depth* -- two independent mechanisms must both allow the operation
* *Architectural impossibility* -- cluster destruction is denied at the admission control layer, not just RBAC
* *Compliance alignment* -- satisfies DISA STIG, NIST, and HIPAA requirements for destructive action controls
* *Operational safety* -- operators can still observe, update, and manage clusters -- only destruction is removed
* *Zero-trust posture* -- even a fully compromised Hub account cannot destroy the fleet

=== Show

NOTE: Ensure you are on the **Hub cluster terminal** and logged into OpenShift. If unsure, run `oc whoami` to verify. See xref:02-details.adoc#_connecting_to_clusters[Connecting to Clusters] for setup.

* Review the custom ClusterRole manifest:
+
[source,role="execute"]
----
cat rbac/rhacm-nondestructive-operator-clusterrole.yaml
----

* Review the ValidatingAdmissionPolicy that provides the second layer of defense:
+
[source,role="execute"]
----
cat rbac/cluster-destruction-prevention-vap.yaml
----
+
Note the key design decisions:
** The ClusterRole omits `delete` for cluster lifecycle resources -- first layer
** The ValidatingAdmissionPolicy denies DELETE operations at the admission control level -- second layer
** `matchConditions` exempts `system:masters`, `cluster-admins`, the `admin` user, and RHACM system service accounts
** All other users are denied, regardless of their RBAC bindings
** Policy and application resources retain `create` -- operators can still deploy governance and workloads

== Part 2 -- Applying the Non-Destructive Role and Admission Policy

=== Know

_The custom ClusterRole replaces the default superset roles for day-to-day operations. The ValidatingAdmissionPolicy provides defense in depth -- even if a user inherits `delete` from a ManagedClusterSet admin binding or any other source, the admission policy blocks the API call._

=== Show

* Apply the non-destructive ClusterRole and binding:
+
[source,role="execute"]
----
oc apply -f rbac/rhacm-nondestructive-operator-clusterrole.yaml
----

* Apply the ValidatingAdmissionPolicy:
+
[source,role="execute"]
----
oc apply -f rbac/cluster-destruction-prevention-vap.yaml
----

* Verify the ClusterRole was created:
+
[source,role="execute"]
----
oc get clusterrole rhacm-nondestructive-operator -o yaml
----

* Verify the ValidatingAdmissionPolicy is active:
+
[source,role="execute"]
----
oc get validatingadmissionpolicy deny-cluster-destruction
oc get validatingadmissionpolicybinding deny-cluster-destruction-binding
----

* Confirm the demo-operator user has the expected non-destructive permissions:
+
[source,role="execute"]
----
oc auth can-i list managedclusters --as=demo-operator
----
+
Expected output: `yes`

+
[source,role="execute"]
----
oc auth can-i update managedclusters --as=demo-operator
----
+
Expected output: `yes`

== Part 3 -- The Destruction Test

=== Know

_This is the most impactful moment of the entire demo. The operator attempts to destroy a managed cluster -- and the Kubernetes API server denies the request. This proves that the protection is not a UI guardrail or a policy warning -- it is an architectural constraint enforced at the admission control level._

=== Show

* **CLI Test** -- Attempt to delete a managed cluster:
+
[source,role="execute",subs="attributes+"]
----
oc delete managedcluster {student_cluster_name} --as=demo-operator --dry-run=server
----
+
Expected output:
+
----
The managedclusters "student-1" is invalid:
  : ValidatingAdmissionPolicy 'deny-cluster-destruction' ... denied the request:
  User demo-operator is denied from deleting managedclusters.
  Cluster destruction is blocked by ValidatingAdmissionPolicy deny-cluster-destruction.
----

* **Hive Test** -- Attempt to delete a ClusterDeployment (if the managed cluster was provisioned via Hive):
+
[source,role="execute",subs="attributes+"]
----
oc delete clusterdeployment {student_cluster_name} -n {student_cluster_name} --as=demo-operator --dry-run=server
----
+
Expected output: Denied by the ValidatingAdmissionPolicy if a ClusterDeployment exists, or `Error from server (NotFound)` if the cluster was imported rather than provisioned via Hive. Both outcomes confirm the operator cannot destroy provisioned clusters.

* **ClusterPool Test** -- Attempt to delete a ClusterPool:
+
[source,role="execute"]
----
oc delete clusterpool demo-pool -n hive --as=demo-operator --dry-run=server
----
+
Expected output: Denied by the ValidatingAdmissionPolicy or `Error from server (Forbidden)`

* **ManagedClusterSet Test** -- Attempt to delete a ManagedClusterSet:
+
[source,role="execute"]
----
oc delete managedclusterset production-clusters --as=demo-operator --dry-run=server
----
+
Expected output: Denied by the ValidatingAdmissionPolicy or `Error from server (Forbidden)`

* **Positive verification** -- Confirm the user CAN still perform non-destructive operations:
+
[source,role="execute"]
----
oc get managedclusters --as=demo-operator
----
+
[source,role="execute",subs="attributes+"]
----
oc label managedcluster {student_cluster_name} demo-verified=true --as=demo-operator --dry-run=server
----
+
Expected output: `managedcluster.cluster.open-cluster-management.io/{student_cluster_name} labeled (server dry run)`

[IMPORTANT]
====
**Deliver this message to the customer:** "Your operator can see every cluster, update labels, manage policies, deploy applications, and observe the entire fleet. The one thing they cannot do is destroy it. This is not a UI warning or an RBAC-only restriction -- it is a ValidatingAdmissionPolicy enforced at the Kubernetes admission control layer. Even if a user inherits `delete` from any ClusterRoleBinding, the admission policy blocks the API call. Even if this account is fully compromised, your fleet is safe."
====

== Key Takeaways

* Defense in depth: the `rhacm-nondestructive-operator` ClusterRole removes `delete` from RBAC, and the ValidatingAdmissionPolicy blocks `delete` at admission control
* Kubernetes RBAC is additive -- ClusterRoles cannot _remove_ permissions granted by other bindings. The ValidatingAdmissionPolicy closes this gap.
* The `delete` verb is blocked for all cluster lifecycle API groups: `managedclusters`, `clusterdeployments`, `clusterclaims`, `clusterpools`, `managedclustersets`
* Protection is enforced at the Kubernetes admission control level -- not a UI guardrail or policy warning
* Operators retain `get`, `list`, `watch`, `update`, `patch` -- only destruction is removed
* RHACM system service accounts are exempted so cluster lifecycle operations continue to function
* This satisfies DISA STIG, NIST, HIPAA, and PCI-DSS requirements for destructive action controls
